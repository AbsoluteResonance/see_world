## 一、SLAM 编译与部署

### 1.1 ORB-SLAM3 编译问题

**Python 版本**：使用 Ubuntu 22.04 自带的 Python 3.10+，系统已安装 OpenCV 4.5.4、Eigen3 3.4.0、Boost 1.74。

**问题：`Examples_old/` 目录缺失**
ORB-SLAM3 最新版用 `--depth=1` clone 时没有 `Examples_old/` 目录，但 CMakeLists.txt 引用了它。新版示例已直接放在 `Examples/` 下。
- **解决**：删除 CMakeLists.txt 中 `Examples_old/` 相关行

**问题：`std::chrono::monotonic_clock` 不存在**
代码使用了 C++11 早期草案中的 `std::chrono::monotonic_clock`，在最终 C++11 标准中被重命名为 `steady_clock`。
- **解决**：全局替换为 `std::chrono::steady_clock`

**问题：Pangolin 需要 C++14**
系统安装的 Pangolin 使用了 `sigslot` 库，依赖 C++14 的 `std::decay_t`、`std::enable_if_t`。ORB-SLAM3 的 CMakeLists.txt 设置 `-std=c++11` 导致编译失败。
- **解决**：升级到 `-std=c++14`

**问题：Pangolin 需要 X Display**
ORB-SLAM3 启动时打开 Pangolin 可视化窗口，在无 GUI 服务器上会 crash。
- **解决**：安装 `xvfb`（虚拟帧缓冲），用 `xvfb-run -a` 运行，或在 Python 接口中捕获 SIGABRT

**问题：关闭时 SIGABRT/SIGSEGV**
ORB-SLAM3 析构时存在内存问题，清理阶段 crash。不会影响输出的轨迹文件。
- **解决**：slam_runner.py 中检查轨迹文件是否存在而非仅看 return_code

### 1.2 VINS-Mono 未部署

Ubuntu 22.04 上无法直接安装 ROS Noetic（依赖 libboost1.71 等旧版库，与 22.04 不兼容）。
- 替代方案：ORB-SLAM3 已有 monocular-inertial 模式，功能等价
- 如需 VINS-Mono：在 Docker 容器中装 ROS Noetic

## 二、Web 前端

### 2.1 上传按钮双击文件选择器

**问题**：上传区域的 `<label for="fileInput">` 和 `dropzone.click → fileInput.click()` 同时触发，导致文件选择器打开两次。
- **解决**：在 dropzone click 处理中检测点击目标，如果是 label 按钮则跳过

### 2.2 上传进度 100% 卡住

**问题**：画廊显示了 `.gitkeep` 隐藏文件，前端加载时出错。
- **解决**：后端 `/api/files` 过滤 `.` 开头的文件

### 2.3 Service Worker 缓存

**问题**：PWA Service Worker 使用 cache-first 策略缓存了 JS 文件，修改代码后浏览器不更新。
- **解决**：更新 `CACHE_NAME` 版本号 + JS 引用加 `?v=N` 参数

## 三、外网隧道

### 3.1 Serveo (弃用)

- 优点：纯 SSH，无需安装
- 缺点：免费版有连接超时限制（~10s），Kimi API 调用（15s）和大文件上传会被掐断

### 3.2 Cloudflare Tunnel (推荐)

- 优点：无带宽限制、无超时限制、自动 HTTPS
- 用法：`cloudflared tunnel --url http://localhost:PORT`
- 注意：免费版 trycloudflare 域名每次重启会变，无 uptime 保证

## 四、视频处理

### 4.1 视频帧提取

用 OpenCV 的 `cv2.VideoCapture` 提取帧，每 10 帧取 1 帧。23MB 视频提了 59 帧。
- 提取的帧需要封装成 TUM 数据集格式（`rgb/` 目录 + `rgb.txt`）
- `rgb.txt` 格式：`timestamp rgb/filename.png`
- 使用 `cv2.CAP_PROP_FPS` 获取真实帧率，计算时间戳

### 4.2 上传改为流式写入

原来用 `await file.read()` 把整个文件读内存再存盘，大文件时占用大量内存且连接易超时。
- **解决**：改为 64KB 分块读取 + 边读边写，超时上限设为 5 分钟
- 上传超时或断连时，清理已部分写入的文件

## 五、后端架构决策

### 5.1 导入路径

- `main.py` 在启动时把 `see_world/` 根目录加入 `sys.path`
- `slam_service.py` 动态导入 `SLAM.python_interface.SLAMRunner`（避免在 uvicorn reload 时重复加载 C++ 模块）
- Python 模块 `SLAM/` 与 `tools/` 同级，通过 `main.py` 的路径注入可达

### 5.2 Job 管理

- 使用内存 dict 存储任务状态（重启后丢失，适合开发阶段）
- 任务状态机：`pending → preparing/extracting → running → completed/failed`
- 前端通过 1s 间隔轮询进度

### 5.3 错误码规范

| 错误码 | 含义 |
|--------|------|
| 0 | 成功 |
| 40001 | 参数错误 |
| 40002 | 文件格式不支持 |
| 40003 | 文件过大 |
| 404 | 资源不存在 |
| 50001 | 服务内部错误 |
| 50002 | Kimi API 调用失败 |
| 50003 | SLAM 模型错误 |
