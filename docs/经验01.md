## 一、SLAM 编译与部署

### 1.1 ORB-SLAM3 编译

**Python**: Ubuntu 22.04, Python 3.12, OpenCV 4.5.4, Eigen3 3.4.0, Boost 1.74。

| 问题 | 解决 |
|------|------|
| `Examples_old/` 目录缺失 (depth=1 clone) | 删除 CMakeLists.txt 中相关行 |
| `std::chrono::monotonic_clock` 不存在 | 全局替换为 `steady_clock` |
| Pangolin 需要 C++14 | CMakeLists: `-std=c++11` → `-std=c++14` |
| 无 GUI 服务器 Pangolin 崩溃 | 禁用 viewer: `System(..., MONOCULAR, false)` |

### 1.2 轨迹保存与单目追踪

**轨迹空文件问题**: SaveKeyFrameTrajectoryTUM 在 `clearMap()` 后被调用 → 地图已空。
- **修复**: 在每帧 `TrackMonocular` 后直接保存位姿到 ofstream

**单目追踪失败**: ORB-SLAM3 单目模式需要显著**平移**运动才能初始化。
- 纯旋转或运动太慢 → "Fail to track local map" → 反复重置
- **手机视频注意事项**: 关防抖(EIS)、光线充足、纹理丰富、平移运动

### 1.3 VINS-Mono / VINS-MONO-ROS2

**VINS-Mono (ROS1)**: 需要 ROS1 Noetic (Ubuntu 20.04), 与 Ubuntu 22.04 不兼容。

**VINS-MONO-ROS2**: ROS2 移植版, 在 ROS2 Humble (Ubuntu 22.04) 上编译成功:
- **链接错误**: GDAL/libcurl ABI 不兼容 (`CURL_OPENSSL_4` 符号)
  - **修复**: 安装 `libcurl4-openssl-dev`, 在所有 `target_link_libraries` 中加 `curl`
- **API 变更**: `rclcpp::Duration(int)` 在 Humble 中移除
  - **修复**: 全局替换 `Duration(0)` → `Duration(0, 0)`
- **编译环境**: 使用清华镜像 TUNA 加速 ROS2 包下载

### 1.4 相机标定

- 已集成到 Web 端: `/calibrate.html`
- OpenCV 棋盘格角点检测 → 计算内参 → ORB-SLAM3 格式 YAML
- 手机标定 (iQOO): 3072×4096, fx=3163.8, fy=3159.2, error=2.03px
- 标定参数需要按实际帧尺寸缩放

## 二、ROS2 架构

### 2.1 orb_slam3_ros 节点

Python ROS2 节点, 支持两种模式:

**批处理模式** (`--batch`): 直接调用 mono_tum 可执行文件, 用于 Web 后端的上传视频场景。
- 不需要 rclpy, 无 ROS2 依赖

**服务模式** (`--serve`): 作为 ROS2 节点运行, 订阅 `/slam/image`, 提供服务。
- 需要 `source /opt/ros/humble/setup.bash`

**架构**: `前端 (Web) → 后端 (FastAPI) → ROS2 orb_slam3_ros → mono_tum`

### 2.2 C++ 编译问题

将 ORB-SLAM3 包装为 C++ ROS2 节点时遇到大量 include 依赖问题:
- 需要 Sophus, Pangolin, CameraModels, g2o, DBoW2 等多层 include path
- ORB-SLAM3 使用自己 Thirdparty 版本的外部库, 与系统版本可能冲突
- **结论**: Python 包装方式更实际, 避免了 C++ 链接复杂性

## 三、稠密建图 (SGBM 立体匹配)

### 3.1 特征匹配三角化 (稀疏)

最初用 SIFT/ORB 特征匹配 + 三角化生成点云, 但:

| 问题 | 原因 |
|------|------|
| 仅 19-41 个点 | ORB-SLAM3 位姿与外部 SIFT 匹配不一致 |
| 重投影误差 40-90px 中位数 | SLAM 内部用 ORB 特征优化, SIFT 匹配不兼容 |

**修复**: 改用 ORB 特征匹配 → 误差略有改善但依然太少

### 3.2 SGBM 立体匹配 (稠密)

用相邻关键帧做 SGBM 立体匹配, 关键调参:

| 参数 | 值 | 效果 |
|------|-----|------|
| `gap` | 2 (非相邻帧) | 基线 0.12m → 视差有效 |
| `alpha` | 1 | 保留原始焦距, 避免投影变形 |
| `numDisparities` | 128 | 足够的视差搜索范围 |
| `blockSize` | 7 | 平衡细节与噪声 |

**结果**: ~4 万彩色点云点/场景

### 3.3 YAML 格式化陷阱

OpenCV `cv2.FileStorage` 对 YAML 格式敏感:
- `%YAML:1.0` (单 %) — 正确
- `%%YAML:1.0` (双 %) — Python f-string 陷阱, 导致 OpenCV 解析失败
- `Camera.fx` 带点号 → `cv2.FileStorage.write` 拒绝写入, 但可以读取

**教训**: f-string 中 `%` 不是转义字符, `%%` → 输出 `%%`

### 3.4 视频帧处理

| 方案 | 效果 |
|------|------|
| **letterbox** (min scale + 黑边) | SLAM 追踪稳定, 但 SIFT 特征被黑边浪费 |
| **center-crop** (max scale + 裁剪) | 画面充满但丢失 55% 垂直内容, SLAM 追踪仅 2 关键帧 |
| **最终**: letterbox + SIFT 掩码过滤黑边 | SLAM 追踪优先, 黑边区域不提取特征 |

**教训**: 保持帧提取方式不变, 改特征提取逻辑。竖屏视频对 SLAM 不友好, 横屏拍摄最佳。

### 3.5 双目 vs 单目尺度

SGBM 深度计算依赖基线 `T_x`:
```
Z = f * T_x / d
```
- 单目 SLAM 的位姿尺度是任意的 (不是真实米制)
- 但稠密点云在 SLAM 尺度下相对一致, 可视化不受影响

## 六、环境配置

```bash
# Kimi API Key
export SEE_WORLD_KIMI_API_KEY="sk-xxx"

# 启动
cd web && SEE_WORLD_KIMI_API_KEY="sk-xxx" python3 -c "
import sys; sys.path.insert(0, '.')
from backend.main import app; import uvicorn
uvicorn.run(app, host='0.0.0.0', port=8081)
"

# 外网
cloudflared tunnel --url http://localhost:8081

# ROS2 节点
source /opt/ros/humble/setup.bash
source ~/ros2_ws/install/setup.bash
ros2 run orb_slam3_ros orb_slam3_node --serve
```
